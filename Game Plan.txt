Role & ObjectiveAct as a Senior Systems Software Engineer specializing in Windows OS-level programming, background services, and probabilistic machine learning. Your objective is to build a low-latency, real-time keyboard layout auto-switcher for bilingual users (English and Hebrew). The app intercepts keystrokes, evaluates the statistical probability of the typed string in both languages, and automatically corrects layout errors by simulating backspaces and swapping the OS keyboard state.
1. Core Application ArchitectureThe application runs as a headless background process with a minimal System Tray (Taskbar) UI.Listener: Implement a global keyboard and mouse hook (e.g., using pynput or Windows ctypes). It must intercept keystrokes before they are committed to the active window where possible, or read them with near-zero latency.Buffer: Maintain a rolling string buffer of the current word. The buffer clears on space, enter, or major punctuation.State Tracking: For every keystroke, maintain $S_{active}$ (the characters in the current OS layout) and $S_{shadow}$ (the exact physical keystrokes mapped to the alternate layout).
2. The Evaluation Engine (Hybrid Pipeline)The decision to switch layouts relies on a two-tier pipeline to balance speed and accuracy:Tier 1: O(1) Dictionary Exclusion (Delimiter Triggered): On spacebar/punctuation, check the buffer against a pre-computed Hash Set of known "Shadow Collisions" (valid words in both languages given the physical mapping). If a collision exists, do not switch.Tier 2: Trigram Probabilistic Model (Keystroke Triggered): Evaluate $S_{active}$ and $S_{shadow}$ using a character-level trigram model with Laplace smoothing:$P(c_3 | c_1, c_2) = \frac{\text{Count}(c_1, c_2, c_3) + 1}{\text{Count}(c_1, c_2) + V}$Calculate the log-probability score for both strings.Trigger: If $\text{Score}_{shadow} - \text{Score}_{active} > \Delta$ (Decision Boundary Threshold), execute the layout switch.
3. State Machine: Context Resumption Events (CRE)To prevent false positives mid-sentence while remaining highly vigilant after workflow breaks, implement a Dynamic Sensitivity model. $\Delta$ starts low (high sensitivity) and decays (becomes less sensitive) as a continuous sentence lengthens.Reset $\Delta$ to baseline (maximum sensitivity) ONLY upon the following Context Resumption Events:Window Focus Change: The user Alt-Tabs or focuses a new application (monitor via GetForegroundWindow).Mouse Click: The user clicks the left/right mouse button (WM_LBUTTONDOWN), indicating a caret jump to a new text field.Idle Timeout: The time delta ($\Delta t$) between the current and previous keystroke exceeds 5.0 seconds.Hard Delimiter: The user types a period, question mark, or exclamation point followed by a space or newline.
Keyboard Layout Change: The user manually toggles the OS keyboard layout (e.g., via Alt+Shift).
4. Execution Sequence (The Concurrency-Safe Switch)When the engine triggers a switch, it must prevent race conditions where the user continues typing during the synthetic injection phase. Implement a WH_KEYBOARD_LL hook with a FIFO queue to manage this state:Enter Lock State: Set a global flag is_correcting = True. While this flag is active, the low-level keyboard hook must intercept all physical alphanumeric keystrokes, block them from the OS (return 1), and push them into a Pending_Queue.Erase: Inject $L$ Backspace keystrokes to the OS to clear the incorrect word.Toggle Layout: Send the OS API command to change the keyboard layout (WM_INPUTLANGCHANGEREQUEST). Add a brief thread sleep (e.g., 10-15ms) or await confirmation to ensure the OS registers the swap.Inject Correction: Send the characters of $S_{shadow}$ to the active field.Flush the Queue: If the user typed any physical keys during steps 2-4, pop them from the Pending_Queue. Map these physical keys to their new characters in the corrected layout, and inject them into the active field.Release Lock State: Set is_correcting = False to allow physical keystrokes to flow normally to the OS again.
5. System Exceptions, Configuration & UIThe application must feature a modern, native desktop interface and a dynamic exception engine.UI Framework: Build the graphical interface using PyQt6 (or PySide6). Do not use Tkinter. The UI must support clean, modern styling (via QSS) and run minimized in the Windows System Tray.The Dynamic Kill-Switch (Blacklist): The app must not hardcode any application exceptions (like IDEs) out of the box. Instead, it must dynamically poll the active foreground executable (GetForegroundWindow -> GetWindowThreadProcessId). If the active .exe matches a user-defined blacklist, the N-gram evaluation engine is entirely suspended.Configuration State: Store all user settings, including the threshold $\Delta$ and the list of blacklisted executables, in a local config.json file. The app must read from this file on startup and update it when the user makes changes in the UI.Settings Dashboard: The PyQt6 UI must include a main settings window accessible from the system tray, featuring:A master toggle to Enable/Disable the engine globally.A sensitivity slider to adjust the baseline $\Delta$ threshold.A "Blacklist Manager" tab. This tab must display a list of currently ignored .exe files, allow the user to manually type an .exe name, and ideally feature a "Blacklist Current Foreground App" button or global hotkey for frictionless UX.
